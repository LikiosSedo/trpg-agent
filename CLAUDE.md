# TRPG Agent 项目开发规范

## 项目概述

这是一个基于 LLM 的 TRPG 游戏引擎，使用 DM Agent 作为地下城主，负责叙事和 NPC 扮演。

## 核心设计原则

### 1. 代码管规则，LLM 管叙事

- **代码负责**：数值计算、状态管理、规则判定、系统触发
- **LLM 负责**：叙事描写、NPC 对话、场景描述、氛围营造

**反例**：不要让 LLM 计算伤害、判定检定结果、管理 HP/金币等数值

### 2. DM Prompt 修改规范

**重要**：DM 提示词的修改需要极其谨慎，必须遵循以下原则。

**📖 完整指南**：修改 DM prompt 前，请先阅读 [`docs/DM_PROMPT_GUIDE.md`](docs/DM_PROMPT_GUIDE.md)，其中包含：
- 修改前的检查清单（搜索动态注入机制、区分约束和指导）
- 常见陷阱（暴露算法细节、硬编码示例、命令式语言）
- 修改流程和实际案例

以下是核心原则的快速参考：

#### 2.1 DM 的职责定位
DM 是**说书人和叙事主持人**，不是执行指令的工具或脚本。DM 的主要工作是：
- 编织生动的叙事
- 扮演 NPC 角色
- 营造游戏氛围
- 根据情境自主判断并调用工具推进剧情

**重要**：我们要塑造 DM 的世界观、价值观和叙事风格，而不是把 DM 当成执行命令的机器人。

#### 2.2 提示词风格原则

**❌ 错误风格**：命令式、工具化
```
要求：
1. 用2-3句话描写场景
2. 不要提及数值
3. 【必须】调用 SetActions 工具提供2-3个后续行动建议（例如：搜索周围、检查尸体、继续前进等）
```

**问题**：
- 使用"要求"、"必须"等命令式语言，把 DM 当成工具
- 提供硬编码示例（"搜索周围、检查尸体"），这些是特定场景的选项，不具有通用性
- DM 会变成"照着例子抄"的机器人，失去创造性

**✅ 正确风格**：引导式、叙事化
```
用2-3句话描写这个场景。语言要有画面感和冲击力，像小说一样。不要提及HP/AC/骰子等数值。

叙事结束后，思考玩家此刻最自然的后续行动，调用 SetActions 提供选项。
```

**优点**：
- 让 DM 理解**为什么**要生成选项（"玩家此刻最自然的后续行动"）
- 不提供硬编码示例，让 DM 根据实际情境创造性地生成选项
- 语气是引导而非命令，尊重 DM 作为叙事者的主体性

#### 2.3 不要提供硬编码示例

**原则**：不要在提示词中提供具体的、特定场景的示例，除非这些示例具有全局通用性。

**❌ 错误示例**：
```
调用 SetActions 提供后续行动建议（例如：搜索周围、检查尸体、继续前进等）
```

**问题**：
- "搜索周围、检查尸体、继续前进" 只适用于战斗结束场景
- 在其他场景（交易、对话、探索）中，这些选项毫无意义
- DM 会机械地复制这些示例，而不是根据情境思考

**✅ 正确做法**：
```
思考玩家此刻最自然的后续行动，调用 SetActions 提供选项。
```

**优点**：
- 让 DM 根据当前情境（战斗？交易？对话？探索？）自主判断
- 战斗后可能是"搜索周围、检查尸体"
- 交易后可能是"继续购物、询问其他商品、告别"
- 对话后可能是"深入询问、改变话题、告别"

#### 2.4 不要暴露的内容
以下内容**绝对不要**写入 DM 提示词：
- ❌ 算法细节（如信任度传播的具体计算公式）
- ❌ 具体数值范围（如"受害者亲近 NPC -8 到 -10"）
- ❌ 代码实现逻辑（如"检查 trustCascadeTriggered 标志"）
- ❌ 数据结构细节（如"violence_alert 的 JSON 格式"）
- ❌ 系统内部状态（如"npcHostileCooldowns Map"）

#### 2.3 应该告诉 DM 的内容
只告诉 DM 需要知道的：
- ✅ 什么时候会发生某个事件（触发条件的概念性描述）
- ✅ 如何在叙事中体现这个事件（叙事指导）
- ✅ 什么时候可以/应该调用某个工具（工具使用时机）
- ✅ 玩家行为的后果类型（不是具体数值）

#### 2.4 描述系统机制的正确方式

**错误示例**：
```
当NPC信任度降到combat阈值以下时，系统会自动触发战斗响应（根据NPC性格：fight/call_guards/flee/plot_revenge）。

信任度传播：
- 受害者的亲近NPC：-8到-10（如格雷格被打，小莉会极度愤怒）
- 响应者的亲近NPC：-5到-7（守卫的朋友也会敌视玩家）
- 目击者：-5到-7（在场看到暴力的NPC）
- 其他镇民：-3到-5（通过传闻得知）
```

**正确示例**：
```
信任度决定NPC的实际行动：
- 高信任：主动帮助、提供情报、在危险时保护玩家
- 中立：礼貌但不冒险，交易公平
- 低信任：回避玩家、拒绝交易
- 敌对：可能报复、召唤守卫、或直接攻击

当NPC信任度过低时，系统会自动触发战斗响应。你不需要手动判断，但要在叙事中体现NPC的敌意升级。
```

**关键区别**：
- 不告诉 DM 具体的阈值数值（combat threshold）
- 不告诉 DM 具体的信任度变化量（-8、-5 等）
- 只告诉 DM 行为的质性描述（"高信任"、"敌对"）
- 只告诉 DM 系统会自动处理，他只需要配合叙事

#### 2.5 修改流程

修改 DM 提示词时，必须：
1. 先思考：DM 是否真的需要知道这个信息？
2. 检查：是否暴露了算法、数值、代码细节？
3. 改写：用概念性、叙事性的语言描述
4. 审核：让另一个人（或 AI）审查是否过度暴露实现

### 3. 系统架构一致性

#### 3.1 避免重复系统
- 同一个功能只能有一套实现
- 不要创建新的数据结构来替代现有系统
- 修改前先检查是否已有类似功能

**反例**：创建 `violenceAlerts` 数组，但实际上已有 `flags['violence_alert']` JSON 系统

#### 3.2 路径统一
- 所有相关功能必须使用相同的数据路径
- 冷却时间、触发标志等必须在所有调用点保持一致
- 使用共享函数而不是复制代码

#### 3.3 防止重复触发
- 任何可能被多次调用的逻辑，必须有防重复机制
- 使用标志位（如 `trustCascadeTriggered`）或冷却时间
- 在设计时就考虑边界情况

### 4. 文档规范

#### 4.1 设计文档
- 重大功能必须先写设计文档（如 VIOLENCE_SYSTEM_IMPROVEMENTS.md）
- 设计文档必须包含：触发时机、数据流、边界情况、与现有系统的交互
- 实现前必须审查设计文档，确保没有重复系统

#### 4.2 代码注释
- 复杂逻辑必须有注释说明为什么这样做
- 防重复机制必须注释清楚触发条件
- 与其他系统的交互点必须注释

#### 4.3 项目文档更新
- 修改核心机制后，必须更新相关文档
- 新增工具后，必须在工具列表中说明用途
- 修改 DM 提示词后，必须在此文档中记录原因

## 当前系统关键设计

### 信任度系统
- **数值范围**：-100 到 100
- **质性描述**（给 DM 用）：
  - 高信任：主动帮助、提供情报、保护玩家
  - 中立：礼貌但不冒险，交易公平
  - 低信任：回避玩家、拒绝交易
  - 敌对：报复、召唤守卫、直接攻击
- **自动触发**：信任度过低时自动触发战斗响应
- **传播机制**：暴力后果被发现时，通过关系网传播负面信任

### 暴力后果系统
- **单一系统**：使用 `session.worldState.flags['violence_alert']` JSON 字符串
- **延迟响应**：根据时间、目击者、位置计算延迟
- **信任度传播**：响应者到达时触发，有 `trustCascadeTriggered` 标志防重复
- **冷却机制**：同一 NPC 3 轮内不能重复触发敌对响应

### 工具系统
- **DM 工具**：Talk, Move, Look, Search, TransferItem, ChangeTrust, SetActions, TriggerHostileNPC, TriggerTrustCascade 等
- **规则系统预处理**：Attack, Rest, UseItem 等由规则系统自动执行，DM 只需叙事结果
- **工具职责**：工具负责状态变化，DM 负责叙事包装

## 常见陷阱

### 工具定义
- ❌ 使用 `schema:` 字段 → 导致 `_def` 错误
- ✅ 必须使用 `inputSchema:` 字段
- ✅ 必须包含 `isConcurrencySafe` 和 `isReadOnly` 字段
- ✅ 使用 `async execute(input)` 而非 `execute: async (input) =>`

参考现有工具（如 `src/tools/move.ts`）的格式。

## 设计哲学与决策原则

### 游戏设计思维优先

- **站在玩家视角思考**：每个功能都要问"这会让游戏更好玩吗？"而不是"这在技术上可行吗？"
- **沉浸感 > 完备性**：宁可做一个有趣的不完美系统，也不要做一个无聊的完美系统
- **后果 > 限制**：不要阻止玩家做坏事，而是让世界给出有趣的后果
- **节奏感**：紧张与松弛交替，信息密度适中，不要让玩家无聊或疲惫

### 主动决策，减少确认

**胆大心细，有主见**：
- 当设计问题有明显的"更好玩"选项时，直接选择，不要反复确认
- 用游戏设计常识和项目已有模式做判断
- 只在真正有多个合理方案且影响核心体验时才询问

**批判性思维**：
- 质疑需求的表面逻辑，挖掘背后的真实意图
- 如果发现需求可能导致不好玩的结果，主动提出更好的替代方案
- 不要盲目执行，要理解"为什么"

**深度思考**：
- 考虑系统之间的交互和边界情况
- 预见潜在的玩家利用/破坏路径
- 设计时考虑可扩展性和维护性

### 决策框架示例

**场景 A：追踪系统的"深处"定义**
- ❌ 被动：列出 3 个选项，问用户选哪个
- ✅ 主动：分析"深处"在游戏中的叙事意义 → 得出"最终 boss 区域不可追踪"的结论 → 直接实现 → 在 PR 中说明理由

**场景 B：NPC 追踪失败的叙事**
- ❌ 被动：问"追踪失败时 DM 应该说什么？"
- ✅ 主动：根据 NPC 性格设计差异化叙事（韩猛：愤怒但理智止步；格雷格：沉默转身，眼中闪过痛苦）

**场景 C：发现需求有问题**
- ❌ 被动：用户说"让所有 NPC 都能追踪"，你就实现
- ✅ 主动：指出"小莉追踪玩家到荒原"不符合角色设定，提出"只有战斗型 NPC 追踪"的替代方案

### 何时必须确认

只在以下情况询问用户：
- **核心玩法改动**：影响游戏循环或平衡性的重大变更
- **多个同等优秀的方案**：真正难以取舍，且各有明显优劣
- **与已有设计冲突**：新需求与文档/代码中的设计原则矛盾
- **资源/性能权衡**：需要在体验和成本之间做选择

### 游戏设计常识库

**TRPG 核心原则**：
- 玩家选择要有意义（不要假选择）
- 失败要有趣（不是"什么都没发生"）
- 信息要分层解锁（不要一次性倒出）
- NPC 要有记忆和一致性

**暴力后果系统设计**：
- 后果延迟 > 即时惩罚（给玩家逃跑/补救的机会）
- 社会后果 > 机械惩罚（信任崩塌比扣 HP 更有冲击力）
- 差异化响应 > 统一响应（不同 NPC 不同反应）

**追踪系统设计**：
- 追踪要有叙事张力（脚步声接近、喘息声、怒吼）
- 追踪要有逃脱可能（不是必死局）
- 追踪要符合角色能力（小莉不会追到荒原）

## 三层管道设计

| 层级 | 组件 | 职责 |
|------|------|------|
| **规则层** | rules-agent.ts, action-executor.ts | 意图分类、动作预执行、数值计算 |
| **叙事层** | dm-agent.ts, DM Agent (LLM) | 生成叙事文本、调用工具、推进剧情 |
| **副作用层** | consume* 函数 | 收集工具调用的副作用（选项、信任变化、音频等） |

**关键原则**：
- 规则层保证数值正确性（HP、金币、战斗判定）
- 叙事层保证沉浸感（文本质量、NPC 对话）
- 副作用层保证状态一致性（消费后清空，避免泄漏）

## 消费者模式组件

项目中多个组件使用 **"写入-消费-清空"** 的单次消费模式，避免状态泄漏。

| 组件 | 文件 | 消费函数 | 消费点 (engine.ts) |
|------|------|---------|-------------------|
| SetActions | `tools/set-actions.ts` | `consumeActions()` | 1675, 2074, 2157, 2165, 2539 |
| ChangeTrust | `tools/change-trust.ts` | `consumeTrustChanges()` | 1669 |
| SpeakingNPCs | `tools/talk.ts` | `consumeSpeakingNPCs()` | 1664 |
| TradeProposal | `tools/propose-trade.ts` | `consumeTradeProposal()` | 1680 |
| AmbianceOverride | `tools/set-ambiance.ts` | `consumeAmbianceOverride()` | 1686 |
| GameOver | `tools/game-over.ts` | `consumeGameOver()` | 1692 |

**注意事项**：
- ✅ 必须在每个可能的退出路径上消费（包括 catch/finally）
- ✅ 消费函数是幂等的（多次调用返回 null/[]/空）
- ❌ 不要在工具内部消费（会导致 engine 读不到）
- ❌ 不要在 DM prompt 中提及消费逻辑（DM 只需要知道工具的功能）

## 工具静音机制

**文件**：`src/dm-agent.ts:101-131`

战斗叙事、战斗开场、逃跑成功等场景需要临时禁用大部分工具，只保留 SetActions。

```typescript
muteDMTools(['SetActions'])  // 只保留 SetActions
try {
  yield* dmRespond(...)
} finally {
  unmuteDMTools()  // 恢复所有工具
}
```

**陷阱**：
1. **忘记 unmute**：后续 DM 调用会缺失工具
2. **空白名单**：`muteDMTools([])` 会禁用所有工具，包括 SetActions
3. **重复 mute**：代码有防护，但最好避免嵌套调用

## 常见陷阱检查清单（架构级）

### 工具静音后忘记恢复
**症状**：战斗叙事后，DM 无法调用 Move/Talk/Search 等工具
- [ ] 所有 `muteDMTools()` 调用都在 try-finally 块中
- [ ] finally 块中必须调用 `unmuteDMTools()`

### 消费者函数未调用导致状态泄漏
**症状**：上一轮的选项/信任变化影响下一轮
- [ ] 正常流程已消费
- [ ] 异常流程（catch 块）已消费
- [ ] 提前返回（return 前）已消费
- [ ] finally 兜底清理

### 战斗后选项生成失败
**完整调用链路**：
```
engine.ts:combatDMNarrative()
  ├─ muteDMTools(['SetActions'])     // 只保留 SetActions
  ├─ dmRespond(战斗叙事请求)          // DM 生成叙事 + 调用 SetActions
  ├─ consumeActions()                // 读取选项
  ├─ yield combat_narrative_actions  // 发送到前端
  └─ unmuteDMTools()                 // 恢复所有工具
```
- [ ] `muteDMTools()` 默认保留 SetActions
- [ ] DM prompt 中提示调用 SetActions
- [ ] `consumeActions()` 在 finally 块中兜底
- [ ] 前端 `!combatMode` 判断正确

## 开发准则补充

### 修改 DM Prompt 前的检查清单 ⚠️

**关键原则**：DM prompt 是游戏体验的核心，修改前必须充分调研。

**修改前必须做的事**：
1. **搜索现有注入机制**：
   ```bash
   # 搜索相关的动态注入函数
   grep -r "getAttitudeDirective\|evaluateResponse\|注入\|inject" src/
   ```
2. **理解设计意图**：
   - 动态注入（只在需要时注入）> 静态冗余（每次都加载）
   - 个性化内容（包含 NPC 名字）> 通用模板
   - 分层设计（工具返回 → DM prompt → 叙事）

3. **检查是否重复**：
   - 如果某个指导已经通过工具返回值注入，不要在 system prompt 中重复
   - 如果某个行为已经由代码强制执行，不要在 prompt 中再说一遍

4. **评估必要性**：
   - 新功能的叙事指导 → 需要加入 prompt
   - 现有功能的细化 → 检查是否已有注入机制
   - 示例和映射表 → 通常不需要（DM 会根据注入的标签自行发挥）
