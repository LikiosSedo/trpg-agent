# TRPG Agent 项目开发规范

## 项目概述

这是一个基于 LLM 的 TRPG 游戏引擎，使用 DM Agent 作为地下城主，负责叙事和 NPC 扮演。

## 核心设计原则

### 1. 代码管规则，LLM 管叙事

- **代码负责**：数值计算、状态管理、规则判定、系统触发
- **LLM 负责**：叙事描写、NPC 对话、场景描述、氛围营造

**反例**：不要让 LLM 计算伤害、判定检定结果、管理 HP/金币等数值

### 2. DM Prompt 修改规范

**重要**：DM 提示词的修改需要极其谨慎，必须遵循以下原则。

**📖 完整指南**：修改 DM prompt 前，请先阅读 [`docs/DM_PROMPT_GUIDE.md`](docs/DM_PROMPT_GUIDE.md)，其中包含：
- 修改前的检查清单（搜索动态注入机制、区分约束和指导）
- 常见陷阱（暴露算法细节、硬编码示例、命令式语言）
- 修改流程和实际案例

以下是核心原则的快速参考：

#### 2.1 DM 的职责定位
DM 是**说书人和叙事主持人**，不是执行指令的工具或脚本。DM 的主要工作是：
- 编织生动的叙事
- 扮演 NPC 角色
- 营造游戏氛围
- 根据情境自主判断并调用工具推进剧情

**重要**：我们要塑造 DM 的世界观、价值观和叙事风格，而不是把 DM 当成执行命令的机器人。

#### 2.2 提示词风格原则

**❌ 错误风格**：命令式、工具化
```
要求：
1. 用2-3句话描写场景
2. 不要提及数值
3. 【必须】调用 SetActions 工具提供2-3个后续行动建议（例如：搜索周围、检查尸体、继续前进等）
```

**问题**：
- 使用"要求"、"必须"等命令式语言，把 DM 当成工具
- 提供硬编码示例（"搜索周围、检查尸体"），这些是特定场景的选项，不具有通用性
- DM 会变成"照着例子抄"的机器人，失去创造性

**✅ 正确风格**：引导式、叙事化
```
用2-3句话描写这个场景。语言要有画面感和冲击力，像小说一样。不要提及HP/AC/骰子等数值。

叙事结束后，思考玩家此刻最自然的后续行动，调用 SetActions 提供选项。
```

**优点**：
- 让 DM 理解**为什么**要生成选项（"玩家此刻最自然的后续行动"）
- 不提供硬编码示例，让 DM 根据实际情境创造性地生成选项
- 语气是引导而非命令，尊重 DM 作为叙事者的主体性

#### 2.3 不要提供硬编码示例

**原则**：不要在提示词中提供具体的、特定场景的示例，除非这些示例具有全局通用性。

**❌ 错误示例**：
```
调用 SetActions 提供后续行动建议（例如：搜索周围、检查尸体、继续前进等）
```

**问题**：
- "搜索周围、检查尸体、继续前进" 只适用于战斗结束场景
- 在其他场景（交易、对话、探索）中，这些选项毫无意义
- DM 会机械地复制这些示例，而不是根据情境思考

**✅ 正确做法**：
```
思考玩家此刻最自然的后续行动，调用 SetActions 提供选项。
```

**优点**：
- 让 DM 根据当前情境（战斗？交易？对话？探索？）自主判断
- 战斗后可能是"搜索周围、检查尸体"
- 交易后可能是"继续购物、询问其他商品、告别"
- 对话后可能是"深入询问、改变话题、告别"

#### 2.4 不要暴露的内容
以下内容**绝对不要**写入 DM 提示词：
- ❌ 算法细节（如信任度传播的具体计算公式）
- ❌ 具体数值范围（如"受害者亲近 NPC -8 到 -10"）
- ❌ 代码实现逻辑（如"检查 trustCascadeTriggered 标志"）
- ❌ 数据结构细节（如"violence_alert 的 JSON 格式"）
- ❌ 系统内部状态（如"npcHostileCooldowns Map"）

#### 2.3 应该告诉 DM 的内容
只告诉 DM 需要知道的：
- ✅ 什么时候会发生某个事件（触发条件的概念性描述）
- ✅ 如何在叙事中体现这个事件（叙事指导）
- ✅ 什么时候可以/应该调用某个工具（工具使用时机）
- ✅ 玩家行为的后果类型（不是具体数值）

#### 2.4 描述系统机制的正确方式

**错误示例**：
```
当NPC信任度降到combat阈值以下时，系统会自动触发战斗响应（根据NPC性格：fight/call_guards/flee/plot_revenge）。

信任度传播：
- 受害者的亲近NPC：-8到-10（如格雷格被打，小莉会极度愤怒）
- 响应者的亲近NPC：-5到-7（守卫的朋友也会敌视玩家）
- 目击者：-5到-7（在场看到暴力的NPC）
- 其他镇民：-3到-5（通过传闻得知）
```

**正确示例**：
```
信任度决定NPC的实际行动：
- 高信任：主动帮助、提供情报、在危险时保护玩家
- 中立：礼貌但不冒险，交易公平
- 低信任：回避玩家、拒绝交易
- 敌对：可能报复、召唤守卫、或直接攻击

当NPC信任度过低时，系统会自动触发战斗响应。你不需要手动判断，但要在叙事中体现NPC的敌意升级。
```

**关键区别**：
- 不告诉 DM 具体的阈值数值（combat threshold）
- 不告诉 DM 具体的信任度变化量（-8、-5 等）
- 只告诉 DM 行为的质性描述（"高信任"、"敌对"）
- 只告诉 DM 系统会自动处理，他只需要配合叙事

#### 2.5 修改流程

修改 DM 提示词时，必须：
1. 先思考：DM 是否真的需要知道这个信息？
2. 检查：是否暴露了算法、数值、代码细节？
3. 改写：用概念性、叙事性的语言描述
4. 审核：让另一个人（或 AI）审查是否过度暴露实现

### 3. 系统架构一致性

#### 3.1 避免重复系统
- 同一个功能只能有一套实现
- 不要创建新的数据结构来替代现有系统
- 修改前先检查是否已有类似功能

**反例**：创建 `violenceAlerts` 数组，但实际上已有 `flags['violence_alert']` JSON 系统

#### 3.2 路径统一
- 所有相关功能必须使用相同的数据路径
- 冷却时间、触发标志等必须在所有调用点保持一致
- 使用共享函数而不是复制代码

#### 3.3 防止重复触发
- 任何可能被多次调用的逻辑，必须有防重复机制
- 使用标志位（如 `trustCascadeTriggered`）或冷却时间
- 在设计时就考虑边界情况

### 4. 数值平衡修改规范 ⚠️

**核心原则**：数值是一个紧耦合的网络。DC、trustMod、初始信任度、怪物属性、物价、经验值之间相互制约，任何单点调整都会在系统中产生涟漪。改数字前必须先理解它在整体平衡中的位置 —— 以"数据平衡师"的审慎态度对待每一个数字。

#### 4.1 哪些数值必须谨慎对待

以下数值**不得擅自调整**，修改前必须和用户讨论：
- 技能检定 DC 值（socialDC / 陷阱 DC / 环境 DC）
- 信任度分段修正（trustMod，如 `trust >= 7 ? -6 : ...`）
- NPC 初始信任度、战斗响应阈值
- 战斗数值（HP / AC / 伤害骰 / 怪物属性 / 角色属性）
- 经济数值（物价、掉落率、金币/经验奖励）
- 信任度/声望变化量（trustDelta 的绝对值）
- 冷却轮数（hostile cooldown、trust cascade 防重复）

#### 4.2 修改数字前必须回答

1. **这个数值目前是多少？它是怎么来的？**（历史决策、参考什么经验值）
2. **它和哪些其他数值耦合？**（如 socialDC 耦合 trustMod + 玩家 CHA 修正 + 熟练加值）
3. **改动后会影响哪些场景？**（列出所有使用它的代码路径和典型玩家场景）
4. **有没有系统级替代方案？**（例如：不改 DC，而改"失败时的语义"）
5. **用户是否同意这个具体数值？**（数字调整必须征得用户同意）

#### 4.3 规则/语义改动 ≠ 数字调整

**✅ 可以直接做（规则/语义层改动）**：
- 改变"失败时是否扣信任"这种**逻辑分支**（persuade 失败 `-1` → `0`）
- 添加新的修正框架（接口/字段），默认参数设为 0 等无副作用值
- 改变数值的**应用时机**（例如只在特定条件下生效）
- 给数字加**上下限 / clamp**（防止数据平衡被边界情况破坏）

**❌ 必须先讨论（数字调整）**：
- 把艾琳娜的 socialDC 从 16 改成 10
- 把 NPC 初始信任从 0 改成 +2
- 把怪物伤害从 1d6 改成 1d8
- 把 trustMod 的 `-6 / -3 / +2 / +4` 梯度改成其他值

#### 4.4 调试输出不是真理

LLM 思考日志、调试输出里的数字（"DC16"、"信任度-1"）只是**当前实现的快照**，不代表它们是正确的。看到不想要的结果时，先评估**这些数字本身是否合理**、以及**规则逻辑是否有问题**，而不是直接调整数字。

**常见陷阱**：用户说"这个 NPC 太难说服了"，不要立刻降 DC；先问"是 DC 定高了？还是玩家的意图根本不该触发说服检定？还是失败的语义过于惩罚？" —— 往往最后一个才是真正的问题。

#### 4.5 改动记录

**数字调整必须记录理由和前后对比**，写在 PR/commit 或本节"修订历史"下方，格式：
```
- 2026-04-XX：[位置] 数值 A → B，理由：...
```
便于以后回溯数据平衡决策。

## 当前系统关键设计

### 信任度系统
- **数值范围**：-10 到 10（`trust-system.ts` 中 clamp 到 [-10, 10]）
- **质性描述**（给 DM 用）：
  - 高信任：主动帮助、提供情报、保护玩家
  - 中立：礼貌但不冒险，交易公平
  - 低信任：回避玩家、拒绝交易
  - 敌对：报复、召唤守卫、直接攻击
- **自动触发**：信任度过低时自动触发战斗响应
- **传播机制**：暴力后果被发现时，通过关系网传播负面信任

#### 信任变化路径（全部入口）

| 路径 | delta | 通道 | 触发 |
|---|---|---|---|
| **正向（对话层）** | | | |
| `Talk(normal)` 自动奖励 | +1 | `dialogue` | 10% 概率 + 3 轮冷却（防刷）|
| `Talk(persuade)` 成功 | +1 | `dialogue` | CHA 说服检定成功 |
| DM 主动 `ChangeTrust(dialogue)` | +1 | `dialogue` | DM 判断玩家展现真诚/共情 **与 normal 自动奖励共享 3 轮冷却锁** |
| **正向（行动层）** | | | |
| 送礼 | 由 DM 决定 | `gift` | `TransferItem` → ChangeTrust |
| 任务完成 | +3 ~ +5 | `quest` | `quest-manager.ts` |
| 兑现承诺 | +1 ~ +2 | `promise` | `trust-system.ts:checkBrokenPromises` |
| **负向（实际坏行为，主要惩罚路径）** | | | |
| 攻击 NPC | -5 | `reputation` | `attack.ts:135` |
| 受害者亲近 NPC（bond ≥ 0.8） | -10 | `reputation` | `propagateViolenceTrust` |
| 受害者亲近 NPC（bond ≥ 0.5） | -8 | `reputation` | `propagateViolenceTrust` |
| 响应者亲近 NPC | -8 ~ -9 | `reputation` | `propagateViolenceTrust` |
| 目击者 | -6 | `witness` | `propagateViolenceTrust` |
| 流言传播（其他镇民） | -4 | `reputation` | `propagateViolenceTrust` |
| 未兑现承诺 | -2 | `promise` | `checkBrokenPromises` |
| **零（显式不扣）** | | | |
| `Talk(persuade)` 失败 | 0 | — | D&D 5e RAW: nothing happens |
| `Talk(deceive)` 成功 | 0 | — | 骗过 ≠ 信任（只是没被拆穿）|
| `Talk(deceive)` 失败 | 0 | — | 失败本身已是惩罚 |
| `Talk(intimidate)` 成功 | 0 | — | 屈服 ≠ 信任提升 |
| `Talk(intimidate)` 失败 | 0 | — | 失败本身已是惩罚 |

**核心原则**：信任下降由"**实际的坏行为**"承担（攻击/偷窃/毁约），**不由"话说得不漂亮"承担**。对话层的失败不扣信任。

#### 关键参数（待调参，记录于此防止后续遗忘）

| 参数 | 位置 | 当前值 | 说明 |
|---|---|---|---|
| `NORMAL_DIALOGUE_TRUST_CHANCE` | `tools/talk.ts` | **10%** | normal 对话自动获得 +1 的概率 |
| `DIALOGUE_TRUST_COOLDOWN_TURNS` | `trust-system.ts` | **3 轮** | dialogue 通道正向信任冷却（normal 10% + DM ChangeTrust 共享）|
| `trustCeiling[chapter]` | `npc-relationships.ts` | NPC 个性化 | 章节信任软上限，超出后 `+delta` 衰减为 `floor(delta/3)` |
| `grudgeTag` 永久仇恨 | `npc-relationships.ts` | NPC 个性化 | 触发后 trust 钉在 -10，正向变化全部拒绝 |

**修改这些数字前**，请先阅读 §4 数值平衡修改规范。这些数字是一个耦合网络,单独调整容易破坏平衡。

#### DM 主动加信任的使用约束

DM 可以调用 `ChangeTrust(channel='dialogue', delta=+1)` 在玩家展现真诚/共情/建设性态度时奖励信任。**但此路径被 3 轮冷却锁保护** —— 超出冷却的调用会被 `trust-system.ts` 静默拒绝(`applied: false`)，防止 DM 滥用。这是**有意的设计**:让 DM 有主动性，但不让 DM 成为"信任水龙头"。

### 暴力后果系统
- **单一系统**：使用 `session.worldState.flags['violence_alert']` JSON 字符串
- **延迟响应**：根据时间、目击者、位置计算延迟
- **信任度传播**：响应者到达时触发，有 `trustCascadeTriggered` 标志防重复
- **冷却机制**：同一 NPC 3 轮内不能重复触发敌对响应

### 工具系统
- **DM 工具**：Talk, Move, Look, Search, TransferItem, ChangeTrust, SetActions, TriggerHostileNPC, TriggerTrustCascade 等
- **规则系统预处理**：Attack, Rest, UseItem 等由规则系统自动执行，DM 只需叙事结果
- **工具职责**：工具负责状态变化，DM 负责叙事包装

## 常见陷阱

### 工具定义
- ❌ 使用 `schema:` 字段 → 导致 `_def` 错误
- ✅ 必须使用 `inputSchema:` 字段
- ✅ 必须包含 `isConcurrencySafe` 和 `isReadOnly` 字段
- ✅ 使用 `async execute(input)` 而非 `execute: async (input) =>`

参考现有工具（如 `src/tools/move.ts`）的格式。

### `public/index.html` 前端单文件陷阱 ⚠️

`public/index.html` 是一个 5600+ 行的**无模块化、无 linter**的单文件,所有函数都挂在同一个全局作用域下。在这个文件里加任何新函数前,**必须先 grep 同名函数**:

```bash
# 在加 function fooBar() {} 之前
grep -n "function fooBar\|fooBar(" public/index.html
```

#### 为什么必须这样做

JavaScript 的 `function` 声明有 **hoisting**,同名函数声明会**静默覆盖**,**没有任何运行时报错或编译警告**。一旦发生:
- 旧函数的所有调用点会执行新函数的代码
- 参数语义错位(旧版 `(npc, npcLocs, isHere)` vs 新版 `(data)`),但 JS 不报错,只会得到 `undefined` 或乱七八糟的结果
- 新函数还会副作用地操作 DOM(比如把弹窗 overlay 强行打开),症状极其诡异且难定位

#### 真实踩过的坑(2026-04 NPC 卡片排队回归)

加 `renderNPCCard(data)` 弹窗内部函数时没 grep,撞上了 line 4217 早就存在的同名 `renderNPCCard(npc, npcLocs, isHere)`(人物面板列表项渲染)。结果:
- 人物面板的 `html += renderNPCCard(...)` 跑了新版 → `html += undefined` → 面板显示 `undefinedundefined`
- 新版函数顺手执行 `overlay.classList.add('open')` → 平白无故弹出一个空白 NPC 卡片
- 弹窗里 `data.title` 恰好读到 NPC 对象的 `title` 字段 → 显示成了某个 NPC 的头衔,误导排查方向

修复:重命名为 `renderNPCCardModal`,并在源码处加注释说明为什么不能用原名。

#### 命名约定(推荐)

为了减少冲突概率,前端函数命名时:
- **弹窗 / 模态相关**:加 `Modal` 后缀(`renderXxxModal`、`showXxxModal`)
- **面板 / 列表项**:用 `renderXxxItem` / `renderXxxRow`
- **新增大功能**:先 `grep -n "function "  public/index.html | sort` 扫一眼现有函数名风格
- **改造旧函数**:如果要把 `foo()` 改成排队/异步版,优先改 `foo` 内部实现而不是新加 `foo2`,保持唯一调用入口

**根本解法**(暂未做):把 `index.html` 拆成多个 ES module 文件,让浏览器/构建工具帮我们抓重名。但拆分工程量大,目前先靠 grep 自律。

## 设计哲学与决策原则

### 游戏设计思维优先

- **站在玩家视角思考**：每个功能都要问"这会让游戏更好玩吗？"而不是"这在技术上可行吗？"
- **沉浸感 > 完备性**：宁可做一个有趣的不完美系统，也不要做一个无聊的完美系统
- **后果 > 限制**：不要阻止玩家做坏事，而是让世界给出有趣的后果
- **节奏感**：紧张与松弛交替，信息密度适中，不要让玩家无聊或疲惫

### 主动决策，减少确认

**胆大心细，有主见**：
- 当设计问题有明显的"更好玩"选项时，直接选择，不要反复确认
- 用游戏设计常识和项目已有模式做判断
- 只在真正有多个合理方案且影响核心体验时才询问

**批判性思维**：
- 质疑需求的表面逻辑，挖掘背后的真实意图
- 如果发现需求可能导致不好玩的结果，主动提出更好的替代方案
- 不要盲目执行，要理解"为什么"

**深度思考**：
- 考虑系统之间的交互和边界情况
- 预见潜在的玩家利用/破坏路径
- 设计时考虑可扩展性和维护性

### 决策框架示例

**场景 A：追踪系统的"深处"定义**
- ❌ 被动：列出 3 个选项，问用户选哪个
- ✅ 主动：分析"深处"在游戏中的叙事意义 → 得出"最终 boss 区域不可追踪"的结论 → 直接实现 → 在 PR 中说明理由

**场景 B：NPC 追踪失败的叙事**
- ❌ 被动：问"追踪失败时 DM 应该说什么？"
- ✅ 主动：根据 NPC 性格设计差异化叙事（韩猛：愤怒但理智止步；格雷格：沉默转身，眼中闪过痛苦）

**场景 C：发现需求有问题**
- ❌ 被动：用户说"让所有 NPC 都能追踪"，你就实现
- ✅ 主动：指出"小莉追踪玩家到荒原"不符合角色设定，提出"只有战斗型 NPC 追踪"的替代方案

### 何时必须确认

只在以下情况询问用户：
- **核心玩法改动**：影响游戏循环或平衡性的重大变更
- **多个同等优秀的方案**：真正难以取舍，且各有明显优劣
- **与已有设计冲突**：新需求与文档/代码中的设计原则矛盾
- **资源/性能权衡**：需要在体验和成本之间做选择

### 游戏设计常识库

**TRPG 核心原则**：
- 玩家选择要有意义（不要假选择）
- 失败要有趣（不是"什么都没发生"）
- 信息要分层解锁（不要一次性倒出）
- NPC 要有记忆和一致性

**暴力后果系统设计**：
- 后果延迟 > 即时惩罚（给玩家逃跑/补救的机会）
- 社会后果 > 机械惩罚（信任崩塌比扣 HP 更有冲击力）
- 差异化响应 > 统一响应（不同 NPC 不同反应）

**追踪系统设计**：
- 追踪要有叙事张力（脚步声接近、喘息声、怒吼）
- 追踪要有逃脱可能（不是必死局）
- 追踪要符合角色能力（小莉不会追到荒原）

## 三层管道设计

| 层级 | 组件 | 职责 |
|------|------|------|
| **规则层** | rules-agent.ts, action-executor.ts | 意图分类、动作预执行、数值计算 |
| **叙事层** | dm-agent.ts, DM Agent (LLM) | 生成叙事文本、调用工具、推进剧情 |
| **副作用层** | consume* 函数 | 收集工具调用的副作用（选项、信任变化、音频等） |

**关键原则**：
- 规则层保证数值正确性（HP、金币、战斗判定）
- 叙事层保证沉浸感（文本质量、NPC 对话）
- 副作用层保证状态一致性（消费后清空，避免泄漏）

## 消费者模式组件

项目中多个组件使用 **"写入-消费-清空"** 的单次消费模式，避免状态泄漏。

| 组件 | 文件 | 消费函数 | 消费点 (engine.ts) |
|------|------|---------|-------------------|
| SetActions | `tools/set-actions.ts` | `consumeActions()` | 1675, 2074, 2157, 2165, 2539 |
| ChangeTrust | `tools/change-trust.ts` | `consumeTrustChanges()` | 1669 |
| SpeakingNPCs | `tools/talk.ts` | `consumeSpeakingNPCs()` | 1664 |
| TradeProposal | `tools/propose-trade.ts` | `consumeTradeProposal()` | 1680 |
| AmbianceOverride | `tools/set-ambiance.ts` | `consumeAmbianceOverride()` | 1686 |
| GameOver | `tools/game-over.ts` | `consumeGameOver()` | 1692 |

**注意事项**：
- ✅ 必须在每个可能的退出路径上消费（包括 catch/finally）
- ✅ 消费函数是幂等的（多次调用返回 null/[]/空）
- ❌ 不要在工具内部消费（会导致 engine 读不到）
- ❌ 不要在 DM prompt 中提及消费逻辑（DM 只需要知道工具的功能）

## 工具静音机制

**文件**：`src/dm-agent.ts:101-131`

战斗叙事、战斗开场、逃跑成功等场景需要临时禁用大部分工具，只保留 SetActions。

```typescript
muteDMTools(['SetActions'])  // 只保留 SetActions
try {
  yield* dmRespond(...)
} finally {
  unmuteDMTools()  // 恢复所有工具
}
```

**陷阱**：
1. **忘记 unmute**：后续 DM 调用会缺失工具
2. **空白名单**：`muteDMTools([])` 会禁用所有工具，包括 SetActions
3. **重复 mute**：代码有防护，但最好避免嵌套调用

## 常见陷阱检查清单（架构级）

### 工具静音后忘记恢复
**症状**：战斗叙事后，DM 无法调用 Move/Talk/Search 等工具
- [ ] 所有 `muteDMTools()` 调用都在 try-finally 块中
- [ ] finally 块中必须调用 `unmuteDMTools()`

### 消费者函数未调用导致状态泄漏
**症状**：上一轮的选项/信任变化影响下一轮
- [ ] 正常流程已消费
- [ ] 异常流程（catch 块）已消费
- [ ] 提前返回（return 前）已消费
- [ ] finally 兜底清理

### 战斗后选项生成失败
**完整调用链路**：
```
engine.ts:combatDMNarrative()
  ├─ muteDMTools(['SetActions'])     // 只保留 SetActions
  ├─ dmRespond(战斗叙事请求)          // DM 生成叙事 + 调用 SetActions
  ├─ consumeActions()                // 读取选项
  ├─ yield combat_narrative_actions  // 发送到前端
  └─ unmuteDMTools()                 // 恢复所有工具
```
- [ ] `muteDMTools()` 默认保留 SetActions
- [ ] DM prompt 中提示调用 SetActions
- [ ] `consumeActions()` 在 finally 块中兜底
- [ ] 前端 `!combatMode` 判断正确

## 开发准则补充

### 修改 DM Prompt 前的检查清单 ⚠️

**关键原则**：DM prompt 是游戏体验的核心，修改前必须充分调研。

**修改前必须做的事**：
1. **搜索现有注入机制**：
   ```bash
   # 搜索相关的动态注入函数
   grep -r "getAttitudeDirective\|evaluateResponse\|注入\|inject" src/
   ```
2. **理解设计意图**：
   - 动态注入（只在需要时注入）> 静态冗余（每次都加载）
   - 个性化内容（包含 NPC 名字）> 通用模板
   - 分层设计（工具返回 → DM prompt → 叙事）

3. **检查是否重复**：
   - 如果某个指导已经通过工具返回值注入，不要在 system prompt 中重复
   - 如果某个行为已经由代码强制执行，不要在 prompt 中再说一遍

4. **评估必要性**：
   - 新功能的叙事指导 → 需要加入 prompt
   - 现有功能的细化 → 检查是否已有注入机制
   - 示例和映射表 → 通常不需要（DM 会根据注入的标签自行发挥）

**案例：信任度系统**
- ❌ 错误：在 prompt 中添加"信任度 → 行为映射表"
- ✅ 正确：`getAttitudeDirective()` 已经在 Talk 工具中动态注入详细的行为指导
- 💡 教训：先搜索 `getAttitudeDirective`，发现已有完善的注入机制
