# DM Prompt 修改指南

本文档指导如何正确地修改 DM 提示词（`src/dm-prompt.ts`），避免常见陷阱。

---

## 核心原则

### 1. DM 是叙事主持人，不是执行工具

**错误思维**：把 DM 当成执行命令的机器人
```
❌ 要求：
1. 用2-3句话描写场景
2. 【必须】调用 SetActions 工具提供2-3个后续行动建议（例如：搜索周围、检查尸体、继续前进等）
```

**正确思维**：塑造 DM 的世界观和叙事风格
```
✅ 用2-3句话描写这个场景。语言要有画面感和冲击力，像小说一样。不要提及HP/AC/骰子等数值。

叙事结束后，思考玩家此刻最自然的后续行动，调用 SetActions 提供选项。
```

**关键区别**：
- 命令式语言（"要求"、"必须"）→ 引导式语言（"思考"、"自然地"）
- 硬编码示例（"搜索周围、检查尸体"）→ 让 DM 根据情境创造
- 告诉 DM "怎么做" → 告诉 DM "为什么"

---

## 修改前的检查清单

### ✅ 第一步：搜索是否已有动态注入机制

很多信息不需要写在 prompt 中，因为系统会在运行时动态注入。

**常见的动态注入机制**：

| 信息类型 | 注入函数 | 注入时机 | 文件位置 |
|---------|---------|---------|---------|
| NPC 态度标签 | `getAttitudeDirective()` | Talk 工具调用时 | `trust-system.ts:156` |
| NPC 可透露的情报 | `getGatedFacts()` | Talk 工具调用时 | `trust-system.ts:206` |
| 章节剧情节点 | `getChapterContext()` | 每轮开始时 | `chapter-manager.ts` |
| 任务提示 | `getQuestHint()` | 每轮结束时 | `quest-manager.ts` |

**如何搜索**：
```bash
# 搜索相关的注入函数
grep -r "getAttitudeDirective\|inject\|注入" src/

# 搜索工具返回值中的内容
grep -r "NPC态度\|NPC上下文" src/tools/
```

**判断标准**：
- 如果某个信息已经通过工具返回值注入 → **不要在 prompt 中重复**
- 如果某个信息是个性化的（包含 NPC 名字、具体数值）→ **应该动态注入，不要硬编码**

---

### ✅ 第二步：区分系统约束和叙事指导

| 类型 | 特征 | 措辞风格 | 示例 |
|------|------|---------|------|
| **系统约束** | 违反会导致技术问题（数据不一致、UI 错乱） | 可以用"必须"、"绝对禁止" | "物品必须通过 TransferItem 工具转移" |
| **叙事指导** | 为了更好的游戏体验 | 应该解释"为什么" | "场景连贯性：每次回应必须承接上一轮的场景和玩家选择。如果玩家在镇口选择进酒馆，下一段必须描写走进酒馆的过程，而不是直接出现在酒馆里。" |

**系统约束示例**（可以用强硬措辞）：
- 不要编造 HP/金币数值（会导致数据不一致）
- 不要在文本中写选项列表（会和 SetActions 重复显示）
- 战斗由系统触发，DM 不能自己发起（会绕过战斗系统）

**叙事指导示例**（应该解释原因）：
- 场景连贯性：避免跳切，让玩家有空间感
- NPC 出场要提全名：系统需要通过名字识别谁在说话
- 信息分层解锁：一次性倒出所有信息会让玩家失去探索乐趣

---

### ✅ 第三步：避免硬编码示例

**原则**：不要在提示词中提供具体的、特定场景的示例，除非这些示例具有全局通用性。

**错误示例**：
```
❌ 调用 SetActions 提供后续行动建议（例如：搜索周围、检查尸体、继续前进等）
```

**问题**：
- "搜索周围、检查尸体、继续前进" 只适用于战斗结束场景
- 在其他场景（交易、对话、探索）中，这些选项毫无意义
- DM 会机械地复制这些示例，而不是根据情境思考

**正确做法**：
```
✅ 思考玩家此刻最自然的后续行动，调用 SetActions 提供选项。
```

**例外情况**（可以提供示例）：
- 工具调用格式示例（展示 JSON 结构）
- 特定场景的反应类型（如 NPC 对暴力事件的反应：震惊、愤怒、恐惧）

## 常见陷阱

### 陷阱 1：暴露算法细节

**错误**：
```
❌ 态度标签的含义（从负到正）：
- **回避**（trust ≤ -6）：NPC看到你立刻离开
- **敌对**（trust -5 到 -3）：充满恶意
- **冷淡**（trust -2 到 -1）：只给最简短回答
...
```

**为什么错**：
- 这是算法细节（具体的数值阈值）
- 系统已经通过 `getAttitudeDirective()` 动态注入详细的态度指令
- 如果修改阈值，需要同步修改两处（代码 + prompt）

**正确做法**：
```
✅ 系统会在 NPC 对话时自动注入态度标签，格式如：
"[NPC态度：深厚信任（trust 8）。格雷格视你为密友或家人，愿意分享核心秘密...]"

根据标签描述来设计 NPC 的对话语气、肢体语言、实际行动。
```

---

### 陷阱 2：重复已有的性格描述

**错误**：
```
❌ 性格一致性（参考 NPC 性格描述）：
- 格雷格：沉默、压抑的愤怒，可能直接动手
- 格罗姆：暴躁地大吼，砰地拍桌，威胁要报告守卫
- 小莉：惊恐地躲到角落，哭泣
```

**为什么可能有问题**：
- 如果前面已经有详细的 NPC 性格描述，这里就是重复
- DM 会机械地复制这些反应，而不是根据性格自主创造

**何时可以保留**：
- 如果这是**特定场景**（如暴力事件）的反应类型指导
- 如果前面的性格描述是通用的，没有针对这个场景的具体反应

**判断标准**：
- 通用性格描述（"格雷格话少句句落地"）→ 不要重复
- 特定场景反应类型（"暴力事件时：格雷格沉默压抑 vs 格罗姆暴躁大吼"）→ 可以保留

---

### 陷阱 3：命令式语言

**错误**：
```
❌ 要求：
1. 先输出叙事文字
2. 然后调用 SetActions 工具
3. 【必须】提供2-3个后续行动建议
```

**为什么错**：
- 把 DM 当成执行指令的工具
- 没有解释"为什么"要这样做
- 限制了 DM 的创造性

**正确做法**：
```
✅ 叙事结束后，思考玩家此刻最自然的后续行动，调用 SetActions 提供选项。
这让玩家能够流畅地继续游戏，而不是陷入"不知道该做什么"的困境。
```

## 修改流程

### 1. 明确修改目的

**问自己**：
- 这个修改是为了修复 bug，还是改善体验？
- 这个信息 DM 真的需要知道吗？
- 这个信息是否已经通过其他方式注入了？

### 2. 搜索相关代码

```bash
# 搜索相关的注入机制
grep -r "关键词" src/

# 搜索工具定义
grep -r "Tool.*name.*关键词" src/tools/

# 搜索系统函数
grep -r "function.*关键词" src/
```

### 3. 理解设计意图

- 阅读相关代码的注释
- 查看 `docs/ARCHITECTURE.md` 了解系统设计
- 查看 `CLAUDE.md` 了解项目规范

### 4. 起草修改

**自查清单**：
- [ ] 是否暴露了算法细节（具体数值、阈值、公式）？
- [ ] 是否提供了只适用于特定场景的硬编码示例？
- [ ] 是否使用了命令式语言（"要求"、"必须"）而没有解释原因？
- [ ] 是否重复了已经动态注入的信息？
- [ ] 是否重复了前面已有的描述？

### 5. 测试验证

- 启动游戏，观察 DM 的实际行为
- 检查 DM 是否机械地复制示例
- 检查 DM 是否理解了"为什么"而不是只记住"怎么做"

## 示例：修改信任度系统说明

### 修改前（有问题）

```markdown
**态度标签的含义**（从负到正）：
- **回避**（trust ≤ -6）：NPC看到你立刻离开，不愿接触
- **敌对**（trust -5 到 -3）：充满恶意，可能威胁、嘲讽或驱赶
- **冷淡**（trust -2 到 -1）：只给最简短回答，语气生硬
- **中立**（trust 0）：礼貌但有距离，按职业规范行事
- **礼貌**（trust 1-2）：有基本好感，正常交流
- **友好**（trust 3-4）：有好感，深入交流，给优惠或方便
- **信任**（trust 5-6）：认可你，透露重要信息，提供实质帮助
- **深厚信任**（trust 7-10）：视你为密友/家人，分享核心秘密

**信息门控**：负值不透露情报，0透露2条，1-2透露4条，3-4透露6条，5-6透露8条，7-10全部透露
```

**问题**：
1. 暴露了具体的数值阈值（≤ -6, -5 到 -3 等）
2. 暴露了信息门控的精确规则（0透露2条，1-2透露4条）
3. 这些信息已经通过 `getAttitudeDirective()` 动态注入了

### 修改后（正确）

```markdown
### 态度标签的使用

系统会在 NPC 对话时自动注入态度标签，格式如：
"[NPC态度：深厚信任（trust 8）。格雷格视你为密友或家人，愿意分享核心秘密、提供无条件帮助...]"

**态度标签是你的行为指导**：
- 标签会告诉你这个 NPC 此刻对玩家的真实感受和行为倾向
- 根据标签描述来设计 NPC 的对话语气、肢体语言、实际行动
- 标签中会说明 NPC 愿意透露哪些信息——按照标签的指导自然地融入对话
- 信任度数值让你微调细节：同样是"友好"，trust 3 和 trust 4 的热情程度略有不同
```

**改进点**：
1. ✅ 不暴露具体阈值，让系统动态注入
2. ✅ 强调"标签是指导"，而不是"背诵映射表"
3. ✅ 解释了标签的作用（为什么要用标签）
4. ✅ 让 DM 理解如何使用标签，而不是记忆规则

---

## 总结

修改 DM prompt 时，记住三个核心问题：

1. **这个信息 DM 真的需要知道吗？**
   - 算法细节 → 不需要
   - 叙事指导 → 需要

2. **这个信息是否已经动态注入了？**
   - 已注入 → 不要重复
   - 未注入 → 可以写在 prompt 中

3. **我是在塑造 DM 的世界观，还是在下命令？**
   - 命令式（"必须"、"要求"）→ 只用于系统约束
   - 引导式（"思考"、"自然地"）→ 用于叙事指导

**记住**：DM 是你的合作伙伴，不是你的工具。给他世界观和价值观，而不是操作手册。
